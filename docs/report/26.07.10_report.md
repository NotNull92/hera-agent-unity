## [13:08] 해결 — CLI 인스턴스 캐시가 컴파일 전 heartbeat를 재사용

- **확정 원인:** `editor refresh --compile`은 `refresh_unity`를 보내기 전 `ready` heartbeat를 조회한다. 이 값이 CLI의 5초 `InstanceCache`에 남은 채 `waitForReady()`에도 재사용되면, 커넥터가 `Heartbeat.MarkCompileRequested()`로 기록한 새 `compiling` 상태를 읽지 않고 이전 `ready` 및 이전 `compileErrors`를 이번 결과로 오인할 수 있다.
- **수정:** `cmd/editor.go`에서 성공한 `refresh_unity` 응답 뒤, `waitForReady()` 폴링 전에 `client.ClearInstanceCache()`를 호출한다. 첫 폴링부터 디스크의 새 heartbeat를 읽는다.
- **회귀 검증:** 캐시된 `ready + CompileErrors=true` → 디스크의 `compiling` → 깨끗한 `ready` 전이를 재현하는 Go 테스트를 추가했다. 수정 전에는 기존 오류를 즉시 반환하며 실패했고, 수정 후 통과했다.
- **실전 확인:** 수정 소스로 `go run . editor refresh --compile --timeout 60000` 실행 후 Unity 상태 `ready`, `console --type error --lines 20`의 `matched=0`을 확인했다.

## [16:26] `editor refresh --compile` 중 idle HTTP 채널에 unsolicited response

- **맥락:** 앞선 play→stop→편집 반복 뒤 `editor refresh --compile` 실행 시.
- **증상(stderr):** `Unsolicited response received on idle HTTP channel starting with "HTTP/1.1 200 OK ... Connection: close"; err=<nil>` 1회 출력. 이후 `refresh_triggered:true` 정상 반환·`status`=ready·`console` 0건.
- **추정:** keep-alive 연결 풀에 앞선 요청(예: play/stop 토글)의 지연 응답이 idle 커넥션에 뒤늦게 도착 → Go net/http가 unsolicited로 경고. 도메인 리로드/에디터 재기동 churn과 연관 가능(같은 세션 앞서 PID 교체 기록됨).
- **영향:** 없음(경고만, 명령 결과 정상). 단 stderr 노이즈가 실패로 오인될 수 있음.
- **권장:** 상태 변경 커맨드 전후로 idle 커넥션을 닫거나(Connection: close 명시) 응답 소진을 보장하면 unsolicited 경고를 줄일 수 있음.

## [19:16] 새 셰이더 임포트 중 `refresh --compile` exit 1 + `console` "connection closed before response"

- **맥락:** GPT가 추가한 `Assets/_Game/UI/Resources/SelectionFlickerWhite.shader`(신규 UI 셰이더) 임포트 검증 — `editor stop` → `editor refresh --compile` 후 `status`=reloading → 재차 `refresh --compile`+`console` 체이닝.
- **증상:** 도메인 리로드/셰이더 임포트가 진행(`status`=compiling)되는 동안, 한 배시 체인의 `editor refresh --compile`이 `{refresh_triggered:true}`를 반환하면서도 **exit code 1**, 뒤이은 `console`은 **`Error: connection closed before response for command: console`**로 실패. in-flight HTTP가 리로드로 끊긴 것.
- **조치/우회:** ~수 초 후 `status`=ready 확인 → `console --type error` 0건, 셰이더 `isSupported=true`(exec)로 정상 임포트 확인. 손실 없음.
- **영향:** 낮음(재시도로 해소). 단 ① 리로드/임포트 중 상태커맨드가 exit 1 + "connection closed"로 실패해 실제 컴파일 실패로 오인될 수 있음, ② §5.3(도메인 리로드 in-flight 드롭)의 또 다른 메시지 변종(앞서 [16:10] no instances, [16:26] unsolicited response에 이은 세 번째 표면화).
- **권장:** `refresh --compile`이 blocking을 표방하는 만큼, 리로드로 커넥션이 끊기면 즉시 exit 1로 반환하지 말고 `ready` 재확인까지 짧게 자동 재연결/재시도 후 최종 상태를 반환하면 오탐을 줄일 수 있음.
